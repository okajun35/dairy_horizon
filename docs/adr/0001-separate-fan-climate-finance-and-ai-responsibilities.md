# ADR-0001: ファン台数、気候・運転期間、財務計算、AI補完の責務を分離する

- Status: Accepted
- Date: 2026-07-15

## Context

Dairy Horizonは、次の情報を一つの意思決定経路へ接続することを目指す。

- 搾乳牛頭数
- 現在のファン台数
- 検討する投資規模
- 将来の暑熱期間
- 設備費と運転費
- 回収に必要な乳量改善
- 一括導入と段階導入の比較

一方、全酪連の計算では、THI、対象頭数、ファン台数は異なる入力として扱われる。THIからファン台数を直接増減すると、気候条件、設備規模、費用計算の責務が混ざり、根拠のない精密化を招く。

また、保存済みの千葉市のCMIP6複数モデルデータは2020〜2034年の日単位データである。2020〜2025年は同一モデルの直近基準、2025〜2034年は将来背景としてファイルを分離する。THI対象日数は算出できるが、日単位データだけから年度別運転時間を直接確定することはできない。

## Decision

### 1. ファン台数

ファン台数は、次のいずれかを根拠として決定論的に扱う。

- 全酪連の頭数基準
- ユーザーが入力した計画台数
- 業者が提示した計画台数

CMIP6やTHIからファン台数を増減させない。頭数基準、ユーザー入力、業者計画は出典区分を分けて保持する。

### 2. CMIP6・THIと運転期間

CMIP6・THIデータは、暑熱対策の対象期間と運転条件の背景を表すために使用する。

- THI対象日数を算出する
- 複数モデルの中央値とモデル範囲を比較する
- 年度別運転時間の入力へ接続する
- ファン台数や投資年を直接決定しない

現在の将来気候データは日単位であるため、運転時間は次のように明示的な入力または仮定を介して算出する。

```text
年間運転時間
= THI対象日数
× 1対象日あたりの運転時間
```

`1対象日あたりの運転時間`は、ユーザー入力、設備制御条件、または明示した標準仮定として保持する。日単位THIから暗黙に推測しない。将来、時間単位の気候データを採用する場合は、この仮定を置き換える別の決定として記録する。

### 3. 気候データの期間境界

取得済みデータの範囲内でのみ比較する。

- 現在相当：2020〜2025年の気象庁・千葉観測を直近観測基準として扱う
- モデル基準：同一CMIP6モデルの2020〜2025年を将来変化量の基準として扱い、観測値とは呼ばない
- 近未来：2026〜2030年のCMIP6複数モデル中央値とモデル範囲
- その先：2031〜2034年のCMIP6複数モデル中央値とモデル範囲
- 2035年以降：未取得と表示し、機械的に外挿しない

期間集計は、各モデルについて期間内の年間THI対象日数を平均し、そのモデル別年平均の中央値・最小・最大を求める。画面では最小〜最大を「モデル範囲」と明記し、最大値だけを高温側の確定ケースとして扱わない。単一モデルを確定予測として表示しない。

観測基準と将来期間を接続するときは、モデルごとに将来期間と2020〜2025年モデル基準の差を求め、その差を気象庁の直近観測基準へ加える。同一モデルを対応させ、集計済み中央値同士を引かない。気象庁観測の欠測日は非暑熱日として扱わず、対象日数を下限・上限で保持する。この観測基準への接続はファン台数や投資年を変更しない。

### 4. 財務計算

財務計算は、ファン計画と気候・運転期間の出力を入力として受け取る決定論的エンジンとする。

```text
設備投資額
= 追加ファン台数 × 1台あたり設備費

年間電力量
= 稼働ファン台数 × 1台あたり消費電力 × 年間運転時間

年間電力費
= 年間電力量 × 電気単価
```

対象期間、設備費、電力費、乳価などを全酪連式へ渡し、回収に必要な防止乳量を計算する。金額と乳量の計算は、テストされたPython関数で行う。

### 5. AI補完

AIは計算エンジンの代わりにはならず、入力負担の軽減とシナリオ設定を担当する。

AIが行ってよいこと：

- 自然文から頭数、台数、費用条件の候補を抽出する
- 不明値へ透明な標準仮定または幅を提示する
- 「電気代が5%上がる」などの条件変更を構造化入力へ変換する
- 計算結果と、結論を変える条件を平易に説明する
- 許可された候補から、次に確認すべき情報を一つ選ぶ

AIが行ってはいけないこと：

- THIから必要ファン台数を発明する
- 設備費、電気代、乳量改善を文章推論だけで計算する
- 取得していない気候期間を生成する
- 決定論的エンジンと異なる数値結果を返す

画面の説明生成では、牛舎・THI・財務の計算済み構造化JSONだけをAPIへ渡す。数値事実はPythonの定型表示とし、AI出力には新しい数値を含めない。API失敗・拒否・検証不合格時は、同じ計算結果を使う定型文へフォールバックする。

「電気代が5%上がる」のように、一度だけ5%上昇するのか毎年5%上昇するのかで結果が変わる場合は、AIが確認するか、採用した解釈を明示する。

## Consequences

### Positive

- 気候予測が変わっても、ファン台数の根拠が追跡できる
- 同じ入力から同じ設備費・電力費・収支分岐点を再計算できる
- 期間別中央値とモデル範囲を同じファン計画で比較できる
- AIによる入力補完と、数値計算の正しさを分離してテストできる
- 2034年を超える未取得期間を、確定値のように表示しない

### Trade-offs

- 日単位の将来THIだけでは運転時間を確定できず、追加の運転時間仮定が必要になる
- THIが悪化しても、必要台数が自動的に増える表示にはならない
- 2035年以降を含む採算期間は、一部未評価または評価開始年の制限が必要になる

## Implementation direction

実装は次の順序で進める。

1. 保存済みCMIP6 JSONから期間別THI対象日数、中央値、範囲を返す純粋関数
2. 対象日数と1日あたり運転時間から年度別運転時間を返す純粋関数
3. ファン台数、消費電力、運転時間、電気単価から年度別電力費を返す純粋関数
4. 設備費、電力費、対象期間を全酪連式へ接続する財務計算
5. AIによるシナリオ入力補完と、決定論的計算結果の説明
